> 所以，当我们利用AI来工作时，第零项工作应该是让AI搞清楚当前项目的最佳实践，哪些实践准则能被AI程序化验收。

## 是的。第零项工作。

**在让AI写任何一行代码之前，先让AI读法。**

不是提示词里的“请遵循最佳实践”，不是系统消息里的“你是一个专家”，不是README里的“我们遵循Airbnb规范”。

**是让AI加载当前项目的验收规约，就像编译器加载语言规范一样。**

## 一、当前AI编程的本质缺陷

我们现在使用AI的方式，是**让一个天才在没有法律的社会里自由发挥**。

| 阶段      | 行为                                       | 问题         |
| --------- | ------------------------------------------ | ------------ |
| **第1步** | 用户说：“帮我写一个登录功能”               | 需求传递     |
| **第2步** | AI根据训练数据中的“常见写法”生成代码       | 依赖统计惯性 |
| **第3步** | 用户review：“这里不用Redux，我们用Zustand” | 事后纠错     |
| **第4步** | AI说：“抱歉，我重新写”                     | 重复劳动     |
| **第5步** | 用户：“算了，我自己改”                     | 信任破裂     |

**这个流程的错误在于：**

AI在**不知道自己身处何种法律体系**的情况下，就开始了产出。

它不知道这个项目：

- 是用函数式还是面向对象？
- 是用CSS-in-JS还是Tailwind？
- 是允许any还是必须strict？
- 是用Jest还是Vitest？
- 是微服务还是单体？
- 是每周发布还是每月发布？

**它只是凭直觉在猜**。

而人类的直觉来自十年经验，AI的直觉来自训练数据中的统计分布。

**问题是：你的项目不是统计分布。**

## 二、第零项工作的完整定义

**第零项工作**：在AI接触任何具体任务之前，先完成**规约加载**。

这不是一个步骤，这是一个**协议层**。

```
[用户/团队]                  [AI]
    |                          |
    |—— 这是你的法律体系 ——→   |
    |                          |  ← 加载规约
    |                          |  验证可执行性
    |                          |  确认理解偏差
    |←—— 收到，确认执行 ——    |
    |                          |
    |—— 现在，开始任务1 ——→   |  ← 在法律框架内生成
```

### 这个协议层必须包含：

**1. 规约的显式化**

不是隐性的“我们团队习惯”，是显式的、可被AI程序化解析的**规约文件**。

```yaml
# .ai-compliance.yaml
version: 1.0.0
project:
  name: ecommerce-platform
  type: nextjs-app

engineering:
  typescript: strict
  testing:
    framework: vitest
    coverage: 80
  state_management: zustand
  styling: tailwind
  api_pattern: tRPC

product:
  doc_required: true
  prd_template: .docs/templates/PRD.md
  acceptance_criteria: gherkin

compliance:
  - rule: no-console-log
    level: error
    auto_fix: true
  - rule: env-vars-documented
    level: warning
    auto_fix: false
```

**2. 规约的可执行化**

每一条规约都必须对应一个**程序化验收断言**。

不是“代码要清晰”，是：

- 函数复杂度 < 10（Cyclomatic complexity）
- 重复代码块 < 3处（Duplicate detection）
- 所有public API有JSDoc（Documentation coverage）

**不能程序化的规约，就是不成熟的规约。** 先搁置，等想清楚了再纳入。

**3. 规约的不可绕过性**

AI必须**在生成时实时遵守**，而不是生成后被动检查。

这不是ESLint的`--fix`，这是**编译器的类型检查**。

你无法生成一个`string`赋值给`number`变量的代码，编译器不让你通过。

同样，AI**无法生成一个违反规约的代码片段**——不是“生成了再标红”，是**根本生不出来**。

## 三、这需要改变AI的产品形态

目前的AI编程工具（Copilot、Cursor、Continue）都是**生成优先，检查在后**。

第零项工作要求的恰恰相反：**规约优先，生成在后**。

这意味着：

**1. 规约成为上下文的一部分**

不是系统消息里的“你是一个专家工程师”，是**显式加载的约束矩阵**。

就像给一个数学家指定公理体系：

- 在欧氏几何里，平行线不相交。
- 在黎曼几何里，平行线相交。

**没有公理体系，数学无法工作。没有规约体系，AI无法可靠工作。**

**2. 生成引擎必须与验收引擎融合**

现在的架构：

```
用户输入 → 大模型 → 代码生成 → 独立检查工具 → 反馈 → 修复
```

第零项工作要求的架构：

```
用户输入 → 规约加载 → 约束感知解码 → 符合规约的代码
                 ↑___________|
                   实时约束反馈
```

这不是一个工具链，这是一个**新架构的编程界面**。

**3. 规约成为共享资产**

当每个项目都有自己的`.ai-compliance.yaml`，**AI就不再是“一个模型”，而是“该项目的法律主体”**。

- 你换一个模型（GPT-4 → Claude 3.5），只要加载同一份规约，产出风格应该一致。
- 你换一个项目（电商平台 → 内部工具），加载不同的规约，AI自动切换行为模式。

**规约是AI行为的固化层，跨越模型、跨越项目、跨越时间。**

## 四、这是农村包围市场的真正突破口

大厂需要复杂的验收体系——48项，跨部门，元验收。

**个人和小团队只需要第零项工作**。

他们不需要完整的“产品工程成熟度雷达图”，他们需要：

- 每次AI写代码，别再给老子用Redux（我们早就不用Redux了）
- 每次AI写API，别再给老子console.log（上线前删不干净）
- 每次AI写文档，别再给老子用英文注释（团队全是中文）
- 每次AI提交PR，别再给老子改配置文件（.env又双叒叕被提交了）

**这些都是第零项工作的子集**。

不是“48项最佳实践”，是**3-5条保命原则**。

不是“组织级验收体系”，是**一个`.ai-compliance.yaml`文件放在项目根目录**。

不是“AI治理框架”，是**让Cursor别他妈再写`any`了**。

## 五、所以验收体系的终极形态是什么？

**不是一个平台，不是一个SaaS，不是一个仪表盘。**

**是一个文件。**

`.ai-compliance.yaml`

放在每个项目的根目录。

AI第一次接触这个项目时，做的第一件事不是“分析代码”，不是“阅读README”，不是“猜测意图”。

**是读这个文件。**

就像编译器读语言规范，就像法官读法典，就像士兵读军规。

**读完，确认，然后说：**

> **“我已了解本项目的法律体系。我将在此框架内执行你的指令。”**
