# 如何建立一套全面的验收体系

**版本**: 1.0  
**日期**: 2026 年 2 月 18 日  
**适用范围**: 代码、非代码、文件、非文件（现实世界）

---

## 一、验收体系的本质

> **验收体系的本质，不是清单，而是"清单的进化机制"。**

### 1.1 什么是验收？

**验收 = 对照标准，检查是否符合。**

| 验收对象 | 标准来源 | 检查方式 |
|----------|----------|----------|
| 代码 | 编码规范、架构原则 | Linter、Code Review |
| 文档 | 文档模板、内容要求 | 人工审查、AI 检查 |
| 流程 | 工作流程、审批链 | 系统日志、时间戳 |
| 现实世界 | 业务指标、用户体验 | 数据监控、用户反馈 |

### 1.2 为什么需要全面的验收体系？

**因为失败从不来自单一维度。**

```
一个成功的产品需要：
✅ 代码质量高          ← 代码验收
✅ 文档清晰完整        ← 文档验收
✅ 流程合规高效        ← 流程验收
✅ 用户真正满意        ← 现实世界验收

一个失败的产品可能：
❌ 代码完美，但需求做错
❌ 文档齐全，但无人执行
❌ 流程规范，但响应缓慢
❌ 数据达标，但用户流失
```

```
                    ▲

│           第 3 层：流程验收                   │

│   核心问题："我们工作得对吗？"               │
└─────────────────────────────────────────────┘
                    ▲
                    │ 驱动
┌─────────────────────────────────────────────┐
│           第 2 层：文档验收                   │
```

### 2.2 各层验收的特征

| 层级 | 可形式化程度 | 自动化程度 | 验收频率 | 反馈周期 |

| 文档 | 中 | 中 | 每次变更 | 分钟级 |
| 流程 | 中低 | 低 | 每日/每周 | 小时级 |

## 三、代码验收

### 3.1 验收对象

- 源代码（.ts, .rs, .java, .py...）
- 配置文件（.yaml, .json, .toml...）
- 测试用例（*.test.*, *.spec.*）
- 构建脚本（Makefile, package.json scripts...）

### 3.2 验收维度

| 维度 | 检查内容 | 工具示例 |
|------|----------|----------|
| **语法** | 有无语法错误 | 编译器、解释器 |
| **规范** | 命名、缩进、风格 | ESLint, Clippy, Checkstyle |
| **类型安全** | 类型使用是否正确 | TypeScript, Rust 编译器 |
| **架构** | 依赖关系、模块边界 | Madge, ArchUnit |
| **测试** | 覆盖率、测试质量 | Vitest, Jest, pytest |
| **安全** | 漏洞、敏感信息 | Snyk, Semgrep, git-secrets |
| **性能** | 时间复杂度、内存使用 | Benchmark 工具 |
| **可维护性** | 代码复杂度、重复率 | SonarQube, CodeClimate |

### 3.3 实施建议

```yaml
# .ai-compliance.yaml
code-compliance:
  linters:
    - engine: eslint
      config: .eslintrc.js
    - engine: prettier
      config: .prettierrc
  
  rules:
    - id: spec-traceability
      description: 代码必须追溯 spec 文档
      level: warning
    
    - id: no-circular-dependency
      description: 禁止循环依赖
      level: error
    
    - id: require-test-file
      description: 每个模块必须有测试
      level: error
  
  thresholds:
    test-coverage: 80%
    max-function-size: 50 行
    max-file-size: 500 行
```

## 四、文档验收

### 4.1 验收对象

- 需求文档（PRD、用户故事）
- 设计文档（架构设计、API 设计）
- 技术文档（README、API 文档）
- 流程文档（SOP、Runbook）
- 复盘文档（Post-mortem、Retrospective）

### 4.2 验收维度

| 维度 | 检查内容 | 检查方式 |
|------|----------|----------|
| **完整性** | 必需章节是否齐全 | 模板匹配 |
| **一致性** | 文档之间是否矛盾 | 交叉引用检查 |
| **时效性** | 文档是否过时 | 最后更新时间检查 |
| **可追溯性** | 需求→设计→代码→测试 | 追溯链检查 |
| **可读性** | 语言是否清晰 | AI 可读性评分 |
| **可执行性** | 步骤是否可操作 | AI 可执行性检查 |

### 4.3 实施建议

```yaml
# .ai-compliance.yaml
document-compliance:
  templates:
    prd:
      required-sections:
        - 背景与目标
        - 用户故事
        - 功能列表
        - 非功能需求
        - 验收标准
        - 非目标（Non-Goals）
    
    design:
      required-sections:
        - 架构概述
        - 接口设计
        - 数据模型
        - 风险与妥协
    
    retrospective:
      required-sections:
        - 发生了什么（事实）
        - 做得好的
        - 需要改进的
        - 行动项（负责人 + 截止日期）
  
  rules:
    - id: doc-freshness
      description: 文档超过 90 天未更新需标记
      level: warning
    
    - id: doc-traceability
      description: 设计文档必须追溯需求文档
      level: error
```

### 4.4 文档验收检查表示例

```markdown
## PRD 验收检查表

- [ ] 背景与目标清晰
- [ ] 用户故事符合 INVEST 原则
- [ ] 验收标准可量化
- [ ] 非目标明确声明
- [ ] 与现有需求无冲突
- [ ] 技术可行性已评估
- [ ] 业务价值已量化
```

## 五、流程验收

### 5.1 验收对象

- 开发流程（需求→设计→开发→测试→发布）
- 协作流程（Code Review、PR 审批）
- 决策流程（技术选型、架构决策）
- 响应流程（故障处理、客户支持）

### 5.2 验收维度

| 维度 | 检查内容 | 检查方式 |
|------|----------|----------|
| **合规性** | 是否遵循定义流程 | 系统日志分析 |
| **效率** | 流程耗时是否合理 | 周期时间分析 |
| **质量** | 流程产出质量 | 返工率、缺陷率 |
| **透明度** | 流程状态是否可见 | 看板完整性 |
| **可追溯性** | 决策是否有记录 | ADR 完整性 |

### 5.3 实施建议

```yaml
# .ai-compliance.yaml
process-compliance:
  workflows:
    development:
      stages:
        - 需求评审（必须产品 + 技术参与）
        - 技术设计（必须有 ADR）
        - 开发（必须有分支）
        - Code Review（必须 1 人批准）
        - 测试（必须自动化通过）
        - 发布（必须有 Release Note）
    
    incident-response:
      stages:
        - 发现（必须记录时间）
        - 响应（必须 15 分钟内）
        - 修复（必须有临时方案）
        - 复盘（必须 48 小时内）
        - 跟进（必须有行动项）
  
  rules:
    - id: pr-review-required
      description: PR 必须经过 Review
      level: error
    
    - id: adr-required
      description: 重大决策必须有 ADR
      level: warning
    
    - id: retrospective-required
      description: 迭代结束必须有复盘
      level: warning
  
  metrics:
    lead-time: < 7 天
    cycle-time: < 3 天
    pr-review-time: < 24 小时
    incident-response-time: < 15 分钟
```

## 七、元验收：对验收体系的验收

### 7.1 为什么需要元验收？

**因为验收体系本身会腐化。**

```
腐化表现：
❌ 规范过时了，但没人发现
❌ 规范被绕过了，但系统不知道
❌ 新实践出现了，但没被纳入
❌ 验收变成形式主义，失去意义
```

### 7.2 元验收检查什么？

| 检查项 | 检查内容 | 检查频率 |
|--------|----------|----------|
| **规范新鲜度** | 规范是否超过 6 个月未更新 | 每月 |
| **规则绕过率** | 某规则被绕过的次数 | 每周 |
| **新实践缺失** | 代码中有新工具但规范无规则 | 每周 |
| **知行一致性** | 文档声明 vs 实际代码偏差 | 每月 |
| **验收有效性** | 验收通过但线上出问题 | 每次事故 |

### 7.3 元验收规则示例

```yaml
# .ai-compliance.yaml
meta-compliance:
  rules:
    - id: paradigm-freshness
      description: 范式必须每 6 个月评审更新
      threshold: 180 天
      level: warning
    
    - id: rule-bypass-rate
      description: 规则绕过率不能超过 30%
      threshold: 30%
      level: warning
    
    - id: emerging-practice-gap
      description: 新实践必须在 30 天内纳入规范
      threshold: 30 天
      level: info
    
    - id: knowledge-practice-gap
      description: 文档与实际偏差不能超过 20%
      threshold: 20%
      level: error
    
    - id:验收体系健康度
      description: 验收体系本身的健康度评分
      metrics:
        - 规范更新频率
        - 规则绕过率
        - 新实践纳入速度
        - 团队满意度
      level: warning
```

## 八、认知七层：验收体系的终极目标

### 8.1 七层认知问题

| 层级 | 认知状态 | 验收体系的角色 |
|------|----------|---------------|
| 1 | 不知道"我不知道" | **探照灯** - 差距可视化 |
| 2 | 知道"我不知道" | **坐标系** - 决策参照系 |
| 3 | 不知道"我知道" | **挖掘机** - 隐性知识显性化 |
| 4 | 知道"我知道" | **对账机** - 知行一致性报告 |
| 5 | 做好了，但不知道如何做好的 | **归因引擎** - 成功因子分解 |
| 6 | 知道最佳实践，但没做到 | **执法者** - 编译级阻断 |
| 7 | 知道而且做到 | **见证者** - 成熟度确认 |

### 8.2 各层验收策略

#### 第 1 层：不知道"我不知道"

```yaml
策略：探照灯
方法：
  - 行业差距可视化
  - 竞品对比分析
  - 技术雷达扫描

输出示例：
"检测到你的项目使用 Redux，但 2025 年 83% 的新项目使用 Zustand"
```

#### 第 2 层：知道"我不知道"

```yaml
策略：坐标系
方法：
  - 提供行业基准
  - 推荐目标值
  - 给出学习资源

输出示例：
"你的测试覆盖率 32%，行业基准：初创 35-45%，成长 50-65%，推荐目标 45%"
```

#### 第 3 层：不知道"我知道"

```yaml
策略：挖掘机
方法：
  - 隐性知识提取
  - 成功模式识别
  - 最佳实践沉淀

输出示例：
"检测到你的构建比同类项目快 63%，原因是配置了 prebuild 清理，建议纳入规范"
```

#### 第 4 层：知道"我知道"

```yaml
策略：对账机
方法：
  - 文档 - 代码对账
  - 声明 - 执行一致性检查

输出示例：
"架构文档声明使用 Jest，实际检测只有 3 次提交包含测试文件，偏差率 90%"
```

#### 第 5 层：做好了，但不知道如何做好的

```yaml
策略：归因引擎
方法：
  - 成功因子分析
  - 相关性分析
  - 分离运气和能力

输出示例：
"项目顺利完成，关键成功因子：代码变更规模 -34%，测试覆盖率 +18%"
```

#### 第 6 层：知道最佳实践，但没做到

```yaml
策略：执法者
方法：
  - 编译级阻断
  - CI 门禁
  - 不可绕过规则

输出示例：
"❌ 测试覆盖率未达标，禁止合并。例外：需要 CTO 批准。"
```

#### 第 7 层：知道而且做到

```yaml
策略：见证者
方法：
  - 持续验证
  - 成熟度确认
  - 降频巡检

输出示例：
"✅ 项目连续 6 个月满足所有合规检查，转为异常监控模式"
```

## 九、实施路线图

### 9.1 阶段一：代码验收（1-3 个月）

```
目标：建立基础代码验收能力

行动项：
□ 选择 Linter 工具（ESLint、Clippy 等）
□ 配置基础规则（规范、类型、测试）
□ 集成到 CI/CD 流程
□ 设置门禁阈值

产出物：
- .eslintrc.js / clippy.toml
- CI 配置文件
- 代码规范文档
```

### 9.2 阶段二：文档验收（3-6 个月）

```
目标：建立文档验收能力

行动项：
□ 定义文档模板（PRD、设计、复盘）
□ 实现文档检查工具
□ 建立文档 - 代码追溯链
□ 设置文档更新提醒

产出物：
- 文档模板库
- 文档检查脚本
- 追溯关系图
```

### 9.3 阶段三：流程验收（6-9 个月）

```
目标：建立流程验收能力

行动项：
□ 定义核心流程（开发、发布、响应）
□ 配置流程检查点
□ 实现流程指标采集
□ 建立流程改进机制

产出物：
- 流程图/SOP
- 流程指标仪表盘
- 流程改进记录
```

### 9.4 阶段四：现实世界验收（9-12 个月）


## 十、验收体系设计原则

### 10.1 核心原则

| 原则 | 说明 | 反模式 |
|------|------|--------|
| **渐进式** | 从代码层逐步扩展到现实世界 | 试图一步到位 |
| **可形式化** | 优先验收可客观判定的项 | 验收主观模糊的项 |
| **自动化** | 能自动则自动，减少人工 | 依赖人工检查 |
| **可进化** | 验收体系本身可被验收 | 验收体系一成不变 |
| **上下文感知** | 区分核心规则和上下文规则 | 一刀切 |

### 10.2 规则设计原则

```yaml
好的规则：
- 明确：无歧义，可客观判定
- 可执行：有工具可检查
- 可修复：有明确的修复方式
- 有价值：违反会导致实际问题
- 可进化：可随时间调整

坏的规则：
- 模糊："代码要简洁"
- 不可检查："设计要优雅"
- 不可修复："架构不合理"
- 无价值："变量必须 3 个字母"
- 僵化："永远不能用 X 技术"
```

### 10.3 阈值设定原则

```yaml
阈值设定：
- 基于行业基准，而非拍脑袋
- 区分错误（error）和警告（warning）
- 允许例外，但例外需审批
- 定期回顾，根据数据调整

阈值示例：
- 测试覆盖率：
  - 核心模块：80%（error）
  - 全项目：60%（warning）
  - 新模块：30%（info，30 天内达标）
```

## 十二、常见陷阱与规避

### 12.1 陷阱一：验收过度

```
症状：
- 规则过多，开发效率下降
- 团队抱怨"什么都做不了"
- 创新被抑制

规避：
- 区分 error 和 warning
- 设置规则数量上限（如 30 条）
- 定期清理低价值规则
- 允许例外审批流程
```

### 12.2 陷阱二：验收不足

```
症状：
- 关键质量问题未被发现
- 线上事故频发
- 技术债务累积

规避：
- 识别关键质量属性
- 设置强制门禁规则
- 定期回顾事故原因
- 补充缺失的验收规则
```

### 12.3 陷阱三：形式主义

```
症状：
- 验收通过，但质量仍差
- 团队学会"绕过"验收
- 验收变成"打勾游戏"

规避：
- 实施元验收
- 检查验收绕过率
- 将验收与实际质量关联
- 定期回顾验收有效性
```

### 12.4 陷阱四：一成不变

```
症状：
- 规范多年未更新
- 新实践无法纳入
- 团队与规范脱节

规避：
- 设置规范评审周期（6 个月）
- 建立新实践纳入流程
- 鼓励团队提出规范改进
- 实施元验收规则
```

### 12.5 陷阱五：脱离业务

```
症状：
- 验收指标与业务目标无关
- 技术团队自嗨
- 业务方不认可验收价值

规避：
- 将验收指标与业务指标关联
- 邀请业务方参与规范制定
- 定期展示验收对业务的贡献
- 验收现实世界结果，而非仅过程
```

## 十四、总结

### 14.1 核心观点

1. **验收体系的本质是"清单的进化机制"**，不是静态清单
2. **全面的验收体系覆盖四层**：代码、文档、流程、现实世界
3. **元验收是验收体系的免疫系统**，防止规范腐化
4. **认知七层是验收体系的终极目标**，从"不知道不知道"到"知道且做到"
5. **验收体系必须可进化**，否则会成为创新的阻碍

### 14.2 行动建议

```yaml
立即行动（本周）：
  - 评估当前验收成熟度
  - 选择 1-2 个最痛的点开始
  - 配置基础 Linter 规则

短期行动（1 个月）：
  - 建立代码验收流程
  - 定义文档模板
  - 集成到 CI/CD

中期行动（3 个月）：
  - 扩展到文档验收
  - 开始流程验收
  - 建立指标采集

长期行动（6-12 个月）：
  - 实现现实世界验收
  - 建立元验收机制
  - 达到 L4/L5 成熟度
```

### 14.3 最后一句

> **验收体系的价值，不在于达到完美，而在于让团队在从"不知道不知道"到"知道且做到"的路上，每一步都有迹可循。**
